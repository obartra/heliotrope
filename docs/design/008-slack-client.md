# 008 - Slack Client

## Context

Heliotrope's core purpose is to keep a Slack profile photo current based on real-world context. This requires storing a per-user Slack API token securely, uploading images to Slack when the resolved avatar changes, and avoiding unnecessary uploads through hash-based deduplication and rate limiting. The Slack token is sensitive credential material that must be encrypted at rest, never logged, and never returned to the client after initial submission.

This document covers the Slack token storage model, the write and read flows for that token, the image upload client, hash-based deduplication, and rate-limit handling.

## Proposal

### Per-user Slack token storage

The Slack token is stored in Firestore at `users/{uid}/secrets/slack`. Firestore rules deny all client reads and writes to the `secrets` subcollection (design doc 001). Only Cloud Functions access this document via the Admin SDK.

The document schema is defined in the architecture doc (000, section "Per-user secrets: Slack token"). This doc refines the encryption to AES-256-GCM specifically (000 says "AES-GCM") and specifies the `tokenHash` as a SHA-256 hex digest for duplicate detection.

### Token write flow (`setSlackToken`)

File: `apps/functions/src/http/setSlackToken.ts`

1. The user enters a Slack API token in the Settings UI (design doc 011).
2. The UI calls `POST /setSlackToken` with the plaintext token in the HTTPS request body. The request includes a Firebase ID token in the `Authorization` header.
3. The function verifies the ID token using `requireAuthedUser` (design doc 001).
4. The function computes SHA-256 of the submitted token and compares it against the existing `tokenHash` (if any). If they match, the function returns early with `{ ok: true, slackTeamId, slackUserId }` from the existing document, avoiding a redundant Slack API call.
5. The function calls Slack's `auth.test` method with the submitted token to verify it is valid and to retrieve `user_id` and `team_id`.
6. If `auth.test` fails, the function returns `{ ok: false, error: "invalid_token" }` with HTTP 400. The plaintext token is not stored.
7. If `auth.test` succeeds, the function encrypts the token using AES-256-GCM with a key derived from the Functions secret `TOKEN_ENCRYPTION_KEY`. The initialization vector (IV) is generated randomly per encryption and prepended to the ciphertext (stored together in `tokenCipher` as a base64 string).
8. The function writes (or overwrites) the document at `users/{uid}/secrets/slack` with the encrypted token, hash, Slack user/team IDs, and timestamps.
9. The function also updates `users/{uid}/profile/singleton` to set `slack.connected: true`, `slack.teamId`, `slack.userId`, and `slack.teamName` (if available from the Slack response).
10. The function returns `{ ok: true, slackTeamId, slackUserId }`. The plaintext token is never included in the response.

### Token read flow

Only the `syncAvatar` and `syncNow` functions read the Slack token. The flow is:

1. Read `users/{uid}/secrets/slack` via Admin SDK.
2. Decode the base64 `tokenCipher` to extract the IV (first 12 bytes) and the ciphertext (remaining bytes).
3. Decrypt using AES-256-GCM with the key derived from `TOKEN_ENCRYPTION_KEY`.
4. Use the plaintext token for the Slack API call.
5. The plaintext token is never logged, never written to any other document, and never included in the decision trace or signals snapshot.

### Encryption details

- **Algorithm:** AES-256-GCM via Node.js `crypto` module.
- **Key derivation:** The `TOKEN_ENCRYPTION_KEY` Functions secret contains a base64-encoded 32-byte key. On function startup, the key is decoded from base64 and used directly. No key derivation function (KDF) is applied in v1 because the key is already a full-entropy 256-bit value generated by the user.
- **IV:** 12 bytes, generated with `crypto.randomBytes(12)` for each encryption. Stored as the prefix of the `tokenCipher` value.
- **Auth tag:** GCM produces a 16-byte authentication tag, appended to the ciphertext by Node.js `crypto.createCipheriv` when using `getAuthTag()`. The tag is included in the stored ciphertext.
- **Storage format:** `tokenCipher` is a base64 encoding of `IV (12 bytes) || ciphertext || auth tag (16 bytes)`.

The architecture doc mentions KMS as an option. For v1, a direct key from `TOKEN_ENCRYPTION_KEY` is used to avoid KMS complexity and cost. KMS can be adopted later by changing the key derivation step without altering the storage format.

### Slack API client

File: `apps/functions/src/slack/client.ts`

Exports two functions:

#### `validateToken(token: string): Promise<{ ok: true, userId: string, teamId: string } | { ok: false, error: string }>`

Calls Slack's `auth.test` endpoint (`https://slack.com/api/auth.test`) with the token in the `Authorization: Bearer` header. Parses the JSON response using Zod. Returns the user and team IDs on success, or an error string on failure.

#### `uploadAvatar(token: string, imageBytes: Buffer, contentType: string): Promise<{ ok: true } | { ok: false, error: string, retryAfterSeconds?: number }>`

Calls Slack's `users.setPhoto` endpoint (`https://slack.com/api/users.setPhoto`) with the image as a multipart form upload. The token is sent in the `Authorization: Bearer` header.

- On success (HTTP 200, `ok: true` in body): returns `{ ok: true }`.
- On rate limit (HTTP 429): reads the `Retry-After` header (seconds) and returns `{ ok: false, error: "rate_limited", retryAfterSeconds }`.
- On auth failure (HTTP 200, `ok: false`, error `"invalid_auth"` or `"token_revoked"`): returns `{ ok: false, error }`. The caller (syncAvatar) writes this to `slackState.lastUploadError`.
- On any other error: returns `{ ok: false, error }` with the Slack error string or a generic message for unexpected responses.

**Required Slack scopes:** `users.profile:read` and `users.profile:write`. These must be configured on the Slack app by the user before generating the token (documented in the README).

### Hash-based deduplication

File: `apps/functions/src/slack/hash.ts`

Exports: `computeImageHash(imageBytes: Buffer): string`

Computes a SHA-256 hex digest of the raw image bytes. This hash is compared against `slackState.lastUploadedImageHash` before every upload attempt.

The deduplication flow (executed in `syncAvatar` and `syncNow`):

1. Resolve the winning image via the resolver.
2. Download the image bytes from Storage.
3. Compute the hash of the downloaded bytes.
4. Read `slackState/singleton`.
5. If `lastUploadedImageHash` matches the computed hash, skip the upload. Record `uploadSkippedReason: "hash-match"` on the decision document.
6. If the hashes differ, proceed with the upload. On successful upload, write the new hash to `slackState.lastUploadedImageHash`.

This prevents redundant Slack API calls when the resolved image has not changed between evaluation cycles. It also handles the case where the user replaces the bytes of an existing image: the new bytes produce a different hash, triggering an upload.

### Rate limiting

Two layers of rate-limit protection:

#### Application-level rate limiting

The user profile contains `scheduler.minSecondsBetweenSlackUploads` (default 300, meaning 5 minutes). Before uploading, `syncAvatar` checks whether `slackState.lastUploadedAt` is within this window. If so, the upload is skipped with `uploadSkippedReason: "rate-limit"`.

This prevents rapid-fire uploads when the resolver's output oscillates between two images due to borderline signal values (for example, temperature fluctuating around a threshold).

#### Slack API rate limiting

If Slack returns HTTP 429, the `uploadAvatar` function reads the `Retry-After` header and returns it to the caller. The caller does not retry within the current invocation. Instead, it logs a warning, writes `uploadSkippedReason: "rate-limit"` to the decision, and relies on the next scheduled run to attempt the upload.

### `slackState/singleton` document

The document shape is defined in the architecture doc (000) and design doc 002. On a successful upload, `lastUploadError` is set to `null` to clear any previous error.

Written only by Cloud Functions via Admin SDK. The client can read this document (Firestore rules allow owner reads on `slackState`) to display connection status and error information on the Dashboard.

## Alternatives considered

The architecture doc (000) discusses and rejects both storing the token as a Functions secret and using Slack OAuth. See the "Alternatives considered" section in 000 for the reasoning.

**Storing the plaintext token (no encryption) since Firestore rules already block client access.** Firestore rules prevent client reads, but they do not protect against a compromised Firebase Admin SDK credential or a Firestore export. Encrypting the token with a separate key adds defense in depth: even if the Firestore data is exposed, the token remains protected as long as `TOKEN_ENCRYPTION_KEY` is not also compromised.

**Using a KDF (PBKDF2 or HKDF) on `TOKEN_ENCRYPTION_KEY` before encryption.** A KDF adds protection if the key material has low entropy (for example, a passphrase). Since the key is a full-entropy 256-bit random value, a KDF would add computational overhead without meaningful security benefit. This can be revisited if the key management approach changes.

**Content-based hash using perceptual hashing (pHash) instead of SHA-256 of raw bytes.** Perceptual hashing would detect visually identical images even after re-encoding or minor modifications. This was rejected because Heliotrope controls the image pipeline end to end: images are uploaded once to Storage and downloaded unchanged for upload to Slack. Byte-level SHA-256 is simpler, faster, and sufficient for detecting whether the same file is being re-uploaded.

**Retry with exponential backoff on Slack 429 within a single function invocation.** Retrying within the function could reduce the delay between resolution and upload. However, Cloud Functions have execution time limits (default 60 seconds for HTTP, 540 seconds for scheduled), and holding a function instance while waiting for a `Retry-After` period (often 30 to 60 seconds) wastes resources. Deferring to the next scheduled run is simpler and stays well within execution limits.

## Open questions

- Should `setSlackToken` call `users.profile.get` in addition to `auth.test` to retrieve the user's Slack display name? The `auth.test` response provides `team` (workspace name) which is stored in `slack.teamName`, but it does not include the user's display name.
- Should the `TOKEN_ENCRYPTION_KEY` be rotated periodically? If so, the encryption code would need to support decrypting with the old key and re-encrypting with the new key. For v1, key rotation is deferred.
- What should happen if the Slack token is revoked between evaluations? Current proposal: `syncAvatar` writes the error to `slackState.lastUploadError`, and the user re-enters the token in Settings. An alternative is to set `slack.connected: false` on the profile automatically, which would stop evaluation attempts until the token is replaced.

## Acceptance criteria

- `setSlackToken` verifies the Firebase ID token and rejects unauthenticated requests with 401.
- `setSlackToken` calls Slack `auth.test` with the provided token and rejects invalid tokens with 400.
- `setSlackToken` encrypts the token with AES-256-GCM using `TOKEN_ENCRYPTION_KEY` before writing to Firestore.
- `setSlackToken` writes `slackUserId` and `slackTeamId` to the secrets document and updates `slack.connected: true` on the user profile.
- `setSlackToken` returns `{ ok: true, slackTeamId, slackUserId }` on success. The plaintext token is never included in any response.
- `setSlackToken` detects duplicate token submission via `tokenHash` comparison and returns early without re-validating.
- `validateToken` correctly parses Slack `auth.test` responses for both success and failure cases.
- `uploadAvatar` sends the image as multipart form data to `users.setPhoto` with the correct authorization header.
- `uploadAvatar` returns `retryAfterSeconds` on HTTP 429 responses.
- `uploadAvatar` returns descriptive error strings for auth failures and other Slack errors.
- `computeImageHash` produces stable SHA-256 hex digests: the same bytes always produce the same hash, and different bytes produce different hashes.
- Hash deduplication prevents Slack upload when `slackState.lastUploadedImageHash` matches the current image hash.
- Application-level rate limiting prevents uploads within `minSecondsBetweenSlackUploads` of the last upload.
- `slackState.lastUploadError` is written on Slack API failures and cleared on the next successful upload.
- All Slack API calls are mocked via `vi.spyOn(globalThis, 'fetch')` in tests. No real Slack API calls in CI.
- Integration test: mocked Slack upload through the `syncNow` flow confirms the full pipeline from token decryption through upload and state update.
- `pnpm typecheck` and `pnpm lint` pass.
- `pnpm test` passes with all new tests included.
